HoboStreamer Setup Guide

Complete deployment guide: OVH Public Cloud + Ubuntu 25.04 + Cloudflare + Let's Encrypt + Nginx — plus OVH plan sizing analysis

OVH Ubuntu 25.04 Cloudflare Let's Encrypt Nginx Node 20

1. Recommended Production Model

For a small or medium public deployment, the safest baseline is:

  • OVH Public Cloud instance in the nearest region to your users
  • Ubuntu 25.04
  • Cloudflare proxy for the main website
  • Nginx on the origin server
  • Let's Encrypt certificate on the origin
  • Cloudflare SSL/TLS mode set to Full (strict)
  • Node 20 LTS for the application runtime
  • systemd for process supervision
  • ufw on the box
  • Only open the extra streaming ports you actively use
For most operators, expose only 80 and 443 first. Add RTMP, JSMPEG, or WebRTC ports later only if you really need them.

2. What HoboStreamer Exposes

The current server in server/index.js provides:

  • HTTP on PORT, default 3000
  • WebSocket upgrades on /ws/chat, /ws/broadcast, /ws/control, /ws/call, and /ws/game
  • Optional RTMP ingest
  • Optional JSMPEG relay
  • Optional WebRTC / mediasoup
  • Persistent data under data/

Typical ports

PortProtocolPurpose
3000TCPInternal Node app (behind Nginx)
80TCPHTTP redirect & Let's Encrypt validation
443TCPHTTPS to Nginx
1935TCPRTMP ingest
9710TCPJSMPEG video relay
9711TCPJSMPEG audio relay
4443TCP/UDPWebRTC entry point
10000-10100UDPmediasoup RTP range
9935TCPHTTP-FLV playback (if enabled)
Do not expose all of them by default. Only open ports for protocols you actively use.

3. OVH Instance Creation

From the OVH "Create an instance" flow:

  • Template: your chosen instance flavor, e.g. B3-16
  • Region: nearest to your users, e.g. Seattle / US-WEST-LZ-SEA-A
  • Image: Ubuntu 25.04
  • SSH key: required
OVH only accepts RSA and ECDSA SSH keys. ED25519 keys are not accepted in this flow.

So for this deployment, generate either:

  • ECDSA P-384 — recommended if OVH accepts it in your project flow
  • RSA 4096 — safest fallback for maximum compatibility

4. Generate an SSH Key for OVH

Run these commands on your local machine, not on the server.

Option A: ECDSA P-384 key

mkdir -p ~/.ssh
chmod 700 ~/.ssh
ssh-keygen -t ecdsa -b 384 -f ~/.ssh/ovh_hobostreamer_ecdsa -C "ovh-hobostreamer"

Option B: RSA 4096 key

mkdir -p ~/.ssh
chmod 700 ~/.ssh
ssh-keygen -t rsa -b 4096 -o -a 100 -f ~/.ssh/ovh_hobostreamer_rsa -C "ovh-hobostreamer"

Lock down the private key

# ECDSA
chmod 600 ~/.ssh/ovh_hobostreamer_ecdsa
chmod 644 ~/.ssh/ovh_hobostreamer_ecdsa.pub

# RSA
chmod 600 ~/.ssh/ovh_hobostreamer_rsa
chmod 644 ~/.ssh/ovh_hobostreamer_rsa.pub

Print the public key to paste into OVH

# ECDSA
cat ~/.ssh/ovh_hobostreamer_ecdsa.pub

# RSA
cat ~/.ssh/ovh_hobostreamer_rsa.pub

Copy the entire single-line public key into the OVH SSH key field.

Test the key fingerprint locally

# ECDSA
ssh-keygen -lf ~/.ssh/ovh_hobostreamer_ecdsa.pub

# RSA
ssh-keygen -lf ~/.ssh/ovh_hobostreamer_rsa.pub

5. Import the SSH Key into OVH

  1. Go to your Public Cloud project
  2. Open the SSH keys area, or paste the key directly during instance creation
  3. Give the key a clear name, e.g. hobostreamer-seattle-prod
  4. Paste the public key line from the .pub file
  5. Create the instance with that key attached
OVH lets you store SSH keys in the control panel and reuse them across regions.

6. Connect to the New Instance

After the instance is created and shows as ready, connect over SSH. Most OVH Ubuntu images use the ubuntu user.

ECDSA example

ssh -i ~/.ssh/ovh_hobostreamer_ecdsa ubuntu@YOUR_SERVER_IP

RSA example

ssh -i ~/.ssh/ovh_hobostreamer_rsa ubuntu@YOUR_SERVER_IP

Optional: SSH config alias

Host hobostreamer-ovh
    HostName YOUR_SERVER_IP
    User ubuntu
    IdentityFile ~/.ssh/ovh_hobostreamer_ecdsa
    IdentitiesOnly yes

Then connect with:

ssh hobostreamer-ovh

7. First-Login Hardening

Run these commands on the server.

Update the box

sudo apt update
sudo apt full-upgrade -y
sudo apt autoremove -y

Install baseline packages

sudo apt install -y \
  git curl wget unzip ca-certificates gnupg \
  nginx ffmpeg ufw fail2ban sqlite3 \
  build-essential python3 python3-pip jq

Optional: create a dedicated app user

sudo adduser --disabled-password --gecos "" hobostreamer
sudo usermod -aG sudo hobostreamer

Then add your public key for that user:

sudo mkdir -p /home/hobostreamer/.ssh
sudo nano /home/hobostreamer/.ssh/authorized_keys
sudo chown -R hobostreamer:hobostreamer /home/hobostreamer/.ssh
sudo chmod 700 /home/hobostreamer/.ssh
sudo chmod 600 /home/hobostreamer/.ssh/authorized_keys

Harden SSH

sudo nano /etc/ssh/sshd_config

Recommended settings:

PermitRootLogin no
PasswordAuthentication no
PubkeyAuthentication yes
ChallengeResponseAuthentication no
UsePAM yes
X11Forwarding no

Reload SSH:

sudo systemctl reload ssh
Do not disable password auth until you have confirmed key login works.

8. Install Node.js 20 LTS

Although package.json allows Node 18+, Node 20 LTS is the recommended production baseline.

curl -fsSL https://deb.nodesource.com/setup_20.x | sudo -E bash -
sudo apt install -y nodejs
node -v
npm -v

Verify the node binary is in /usr/bin

On some Ubuntu versions the NodeSource package installs to /usr/local/bin or doesn't create a /usr/bin/node symlink. The systemd service and npm scripts need node on the default PATH.

which node            # should print /usr/bin/node
ls -l /usr/bin/node   # should exist

If which node returns a different path or nothing, create a symlink:

sudo ln -sf "$(which node)" /usr/bin/node
Without this symlink you'll see /usr/bin/env: 'node': No such file or directory when systemd starts the service.

9. Clone HoboStreamer

cd /opt
sudo git clone https://github.com/HoboStreamer/HoboStreamer.com.git hobostreamer
sudo chown -R $USER:$USER /opt/hobostreamer
cd /opt/hobostreamer
npm install

If native modules later need rebuilding:

npm rebuild
If mediasoup fails to compile, browser-native WebRTC broadcasting won't be ready until that's fixed.

11. Create Runtime Directories

The app creates them on boot, but you can pre-create them:

cd /opt/hobostreamer
mkdir -p data/vods data/clips data/media data/thumbnails data/emotes data/avatars

12. Bootstrap the App

Initialize the database if needed:

cd /opt/hobostreamer
npm run init-db

Or just start it once manually:

npm start

Then stop it and move on to systemd setup.

13. Cloudflare DNS Setup

Main website

Create these records in Cloudflare DNS:

  • A record for @ → YOUR_OVH_PUBLIC_IP
  • CNAME for www → your apex hostname

Set both to Proxied (orange cloud) after the origin is working.

Optional: direct hostnames for non-HTTP protocols

Only create these if you really need them:

RecordTypeValueProxy
rtmpAYOUR_OVH_PUBLIC_IPDNS only
webrtcAYOUR_OVH_PUBLIC_IPDNS only
jsmpegAYOUR_OVH_PUBLIC_IPDNS only
These are usually DNS only (grey cloud) because they are not standard proxied website traffic.

14. Let's Encrypt on Ubuntu 25.04

For end-to-end encryption with Cloudflare, the origin must present a valid certificate. Cloudflare's Full (strict) mode requires the origin cert to be unexpired, valid for the hostname, and issued by a publicly trusted CA.

Install Certbot

sudo apt install -y snapd
sudo snap install core
sudo snap refresh core
sudo snap install --classic certbot
sudo ln -sf /snap/bin/certbot /usr/local/bin/certbot
certbot --version

Temporary Nginx HTTP site

Before requesting the cert, create a plain HTTP Nginx site so your domain answers on port 80.

sudo nano /etc/nginx/sites-available/hobostreamer
server {
    listen 80;
    listen [::]:80;
    server_name your-domain.example www.your-domain.example;

    location / {
        proxy_pass http://127.0.0.1:3000;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    location /ws/ {
        proxy_pass http://127.0.0.1:3000;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_read_timeout 3600;
    }
}

Enable it:

sudo ln -sf /etc/nginx/sites-available/hobostreamer /etc/nginx/sites-enabled/hobostreamer
sudo rm -f /etc/nginx/sites-enabled/default
sudo nginx -t
sudo systemctl reload nginx

Request the certificate

If Cloudflare proxying causes validation issues, temporarily set DNS records to DNS only until issuance succeeds, then proxy them again.
sudo certbot --nginx -d your-domain.example -d www.your-domain.example

Verify auto-renewal

sudo systemctl list-timers | grep certbot || true
sudo certbot renew --dry-run

15. Cloudflare End-to-End Encryption

Once the origin has a valid Let's Encrypt cert:

  1. Open the Cloudflare dashboard
  2. Go to SSL/TLS
  3. Set encryption mode to Full (strict)

This gives you encrypted browser-to-Cloudflare and encrypted Cloudflare-to-origin traffic, with origin certificate validation.

Recommended Cloudflare settings

  • Always Use HTTPS — enabled
  • Automatic HTTPS Rewrites — enabled
  • WAF / bot protections (if available on your plan)
  • Rate limiting for sensitive paths (if available)
For extra origin protection, add Cloudflare IP allowlists at the firewall or Nginx layer, or configure Authenticated Origin Pulls.

Optional: custom log format with Cloudflare info

sudo nano /etc/nginx/conf.d/log_format_cloudflare.conf
log_format cloudflare_combined '$remote_addr - $remote_user [$time_local] '
                                '"$request" $status $body_bytes_sent '
                                '"$http_referer" "$http_user_agent" '
                                'cf_ray=$http_cf_ray cf_ip=$http_cf_connecting_ip';

Apply it in your site config with:

access_log /var/log/nginx/hobostreamer.access.log cloudflare_combined;
Why this matters: HoboStreamer uses Express with trust proxy enabled. Correct Nginx real-IP handling ensures logs, moderation, rate limiting, and security signals use the real visitor IP instead of a Cloudflare edge IP.

17. Final Nginx HTTPS Config

After Certbot runs, tighten the site config:

server {
    listen 80;
    listen [::]:80;
    server_name your-domain.example www.your-domain.example;
    return 301 https://$host$request_uri;

server {

18. Firewall (UFW)

Minimal website-only deployment

sudo ufw allow OpenSSH
sudo ufw allow 80/tcp
sudo ufw allow 443/tcp
sudo ufw enable
sudo ufw status verbose

If you need RTMP ingest

sudo ufw allow 1935/tcp

If you need JSMPEG

sudo ufw allow 9710/tcp
sudo ufw allow 9711/tcp

If you need WebRTC / mediasoup

sudo ufw allow 4443/tcp
sudo ufw allow 4443/udp
sudo ufw allow 10000:10100/udp
Only open what you actively use.

19. OVH Security Groups

OVH's OpenStack-based Public Cloud default security groups can allow broad traffic by default. Do not rely on OVH defaults alone.

Use at least one of these:

  • ufw on the instance
  • OVH/OpenStack security groups (if you manage networking that way)
  • Nginx allowlists for sensitive origin paths
If you build custom OVH security groups, follow the same principle: start from least privilege and only open the exact ports you need.

20. systemd service

Create a service unit so the app starts on boot and restarts on crash:

sudo nano /etc/systemd/system/hobostreamer.service

Paste the following unit file:

[Unit]
Description=HoboStreamer
After=network.target

[Service]
Type=simple
User=ubuntu
WorkingDirectory=/opt/hobostreamer
Environment=NODE_ENV=production
Environment=PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin
ExecStart=/usr/bin/npm start
Restart=always
RestartSec=5

[Install]
WantedBy=multi-user.target
If you created a dedicated hobostreamer user, change User=ubuntu to User=hobostreamer and ensure ownership of /opt/hobostreamer matches.

Enable and start:

sudo systemctl daemon-reload
sudo systemctl enable hobostreamer
sudo systemctl start hobostreamer
sudo systemctl status hobostreamer

View logs:

journalctl -u hobostreamer -f

21. Close registration after bootstrap

After you create your initial admin and trusted test accounts, close public registration. This is one of the cheapest and strongest anti-abuse controls in the project.

cd /opt/hobostreamer
sqlite3 data/hobostreamer.db "UPDATE site_settings SET value='false' WHERE key='registration_open';"
sqlite3 data/hobostreamer.db "SELECT key, value FROM site_settings WHERE key='registration_open';"

22. Optional fail2ban baseline

Install:

sudo apt install -y fail2ban

Create a basic jail file:

sudo nano /etc/fail2ban/jail.local

Example:

[sshd]
enabled = true
maxretry = 5
findtime = 10m
bantime = 1h

Then:

sudo systemctl enable fail2ban

sudo fail2ban-client status
If most web traffic is behind Cloudflare, make sure your Nginx real-IP config is correct before relying on log-based web jails.

23. Protocol-specific exposure guidance

RTMP

Expose 1935/tcp only if you need OBS or FFmpeg ingest.

JSMPEG

Expose 9710/tcp and 9711/tcp only if you use the relay.

WebRTC

Expose only the exact WebRTC TCP/UDP and RTP ports you need. Make sure:

  • MEDIASOUP_ANNOUNCED_IP matches your public OVH IP
  • Your firewall allows the RTP range
  • Your Cloudflare setup is not expected to proxy raw media ports

24. Operational checks

App health

Verify:

  • http://127.0.0.1:3000/api/health on the server
  • https://your-domain.example/api/health through Nginx and Cloudflare

TLS

  • Browser sees a valid certificate
  • Cloudflare SSL/TLS mode is Full (strict)
  • No redirect loops
  • No mixed-content warnings

Visitor IP restoration

Check Nginx logs and confirm the client IP is not just a Cloudflare edge IP.

Port audit

Check that only intended ports are reachable from the internet:

sudo ss -tulpn
sudo ufw status verbose

25. Full command summary

If you want the short version, the high-level order is:

  1. Generate an OVH-compatible SSH key locally
  2. Create the OVH Ubuntu 25.04 instance with that public key
  3. SSH into the new instance
  4. Update packages and install Nginx, Node 20, FFmpeg, ufw, and fail2ban
  5. Clone HoboStreamer into /opt/hobostreamer
  6. Configure .env
  7. Start the app once and verify /api/health
  8. Set Cloudflare DNS to the OVH IP
  9. Configure Nginx
  10. Issue a Let's Encrypt certificate with Certbot
  11. Enable Cloudflare Full (strict)
  12. Add Cloudflare real-IP config in Nginx
  13. Enable systemd for HoboStreamer
  14. Close public registration after bootstrap
  15. Open only the extra media ports you actually need

26. Security reminders

  • Never use ED25519 if the OVH flow in your account rejects it
  • Keep your private SSH key only on trusted admin machines
  • Disable password SSH once key auth is confirmed
  • Keep JWT_SECRET long and random
  • Keep the website behind Cloudflare
  • Do not expose raw media ports unnecessarily
  • Keep upload limits low on small servers
  • Close registration when not actively onboarding users
  • Monitor data/ growth for VODs, clips, thumbnails, avatars, and emotes

28. Setup summary

For OVH + Ubuntu 25.04, the best HoboStreamer deployment pattern is:

  • RSA or ECDSA SSH key uploaded to OVH
  • Ubuntu 25.04 on OVH Public Cloud
  • Node 20 LTS
  • Nginx reverse proxy on the box
  • Let's Encrypt certificate on the origin
  • Cloudflare proxy with Full (strict)
  • Cloudflare real-IP restoration in Nginx
  • systemd, ufw, fail2ban
  • Only one streaming protocol exposed at a time unless you have a real need for more

OVH Public Cloud Sizing Guide

Executive summary

Best overall pick — B3-32

  • 32 GB RAM — enough for Node, SQLite, WebSockets, chat, thumbnails, clips, and moderate media workload
  • 8 vCores — sane floor for a mixed single-box deployment
  • 200 GB NVMe — much more realistic than 50–100 GB once VODs and clips start accumulating
  • 2 Gbit/s — leaves real headroom for 200 viewers at practical streaming bitrates

Best budget / staging pick

  • B3-8 — cheapest decent general-purpose box
  • C3-8 — cheaper CPU-heavier test box (only 100 GB storage)

Best safer production pick — B3-64

  • 16 vCores is much healthier for 200 viewers with WebRTC, call features, game traffic, clip creation
  • 4 Gbit/s helps if bitrate climbs or you add more streams / cameras / rooms later
The older B2 and C2 families mostly lose on price/performance against the new B3 and C3 families. C2 is the least attractive family for this workload.

What limits HoboStreamer on a single OVH instance

The bottleneck is not just RAM. The main constraints are:

  1. Outbound bandwidth — the first hard wall
  2. CPU — especially for WebRTC
  3. Disk space — once VODs are enabled
  4. Disk I/O
  5. Operational headroom for Node, FFmpeg, thumbnails, clips, chat, and WebSockets

Bandwidth is the first hard wall

For all three playback methods, the origin server pays outbound bandwidth per viewer:

  • JSMPEG: server relays the live stream to every viewer
  • RTMP / HTTP-FLV: server still pays outbound per viewer
  • WebRTC SFU: server forwards media to each viewer; outbound scales with audience
Cloudflare protects the website and proxied HTTP traffic, but raw media ports and direct real-time traffic are not magically offloaded just because the website uses Cloudflare.

CPU becomes more important for WebRTC

  • JSMPEG: usually more bandwidth-bound than CPU-bound
  • RTMP ingest + direct playback: also heavily bandwidth-bound
  • WebRTC / mediasoup: usually more CPU-sensitive (SFU packet forwarding, transport handling, encryption overhead, more real-time state)

Storage dominates once VODs are enabled

The codebase stores locally: SQLite database, VODs, clips, thumbnails, emotes, avatars, and media assets. The OS and app don't need much space — recorded video does.

Bandwidth math for 200 viewers

A useful first-order formula:

required egress Mbps ≈ viewer count × average delivered Mbps per viewer

For 200 viewers:

  • At 1.5 Mbps per viewer: ~300 Mbps raw egress
  • At 2.5 Mbps per viewer: ~500 Mbps raw egress
  • At 4.0 Mbps per viewer: ~800 Mbps raw egress

Real deployments need headroom for TLS/protocol overhead, chat/API/thumbnails, bitrate spikes, reconnections, and background jobs. Target around 70% of the listed port speed as usable sustained capacity.

What that means for 200 viewers

  • 250 Mbit/s plans — not enough except for very low bitrates
  • 500 Mbit/s plans — can work for low-mid bitrate, but not comfortable
  • 1 Gbit/s plans — viable if you keep average delivered bitrate moderate
  • 2 Gbit/s plans — where the setup gets comfortable
  • 4 Gbit/s plans — stop thinking about bandwidth, start thinking about CPU and storage

Storage math for VODs and clips

Rule of thumb: 1 Mbps ≈ 0.44 GB/hour

  • 1.5 Mbps ≈ 0.66 GB/hour
  • 2.5 Mbps ≈ 1.10 GB/hour
  • 4.0 Mbps ≈ 1.76 GB/hour
  • 6.0 Mbps ≈ 2.64 GB/hour

Usable media space after OS/app reserve (~15–25 GB)

Approximate retained recording hours (at 2.5 Mbps)

  • 30 GB usable → ~27 hours
  • 80 GB usable → ~73 hours
  • 180 GB usable → ~164 hours
  • 380 GB usable → ~345 hours

At 4.0 Mbps

  • 30 GB usable → ~17 hours
  • 80 GB usable → ~45 hours
  • 180 GB usable → ~102 hours
  • 380 GB usable → ~216 hours
Storage takeaway: 50 GB is test-only. 100 GB is okay for dev/alpha. 200 GB is the first production-usable size. 400 GB is much better but still not archival. Long-term, push older VODs to object storage.

Streaming method plan fit

WebRTC / mediasoup

The method most likely to become CPU-sensitive first.

  • B3-32 for moderate production
  • B3-64 for safer 200-viewer production
  • C3-32 only if you are convinced CPU will matter more than RAM/bandwidth economics

If your expected audience is up to 200 and WebRTC is a real primary path, start at B3-64 — the full balance of 16 vCores + 4 Gbit/s + 400 GB NVMe is much healthier.

JSMPEG

More straightforward on the server side — becomes bandwidth-bound quickly.

  • B3-16 for smaller audiences and testing
  • B3-32 for up to 200 viewers at moderate bitrates
  • B3-64 if you want bitrate headroom or larger audience spikes

For a JSMPEG-heavy public deployment, B3-32 is the sweet spot.

RTMP ingest + direct playback

RTMP ingest itself is light. Viewer delivery is what costs you — again outbound bandwidth and storage.

  • B3-16 for low-mid audience counts
  • B3-32 for a real public rollout
  • B3-64 if you expect higher bitrate, spikes, or multiple simultaneous live loads

Best credit strategies

Strategy A — longest useful beta: Run B3-8 or C3-8 during buildout, staging, and feature validation. Upgrade to B3-32 only when audience and VOD usage justify it.
Strategy B — realistic public beta: Start on B3-16. Accept that $200 covers only ~82 days. Resize to B3-32 when you need the larger envelope.
Strategy C — short burst launch: Use B3-32 for launch testing. Know that $200 gives ~41 days continuously.

Recommended picks by scenario

Scenario A — dev, staging, private alpha

Pick: B3-8 — cheapest balanced plan. Enough for app setup, testing, internal streams, chat, docs, and smoke validation. Credit lasts the full 90 days easily.

Scenario B — public beta, one main stream, realistic growth

Pick: B3-16 (budget) or B3-32 (best value). Compared with B3-16, B3-32 gives 2× RAM, 2× vCPU, 2× disk, 2× network for exactly 2× price — the operational breathing room is much better.

Scenario C — target up to 200 concurrent viewers on one box

Pick: B3-32 (minimum serious) or B3-64 (safer). For 200 viewers, the question becomes: what is your average delivered bitrate? How WebRTC-heavy is the workload? How much VOD retention locally? B3-32 is the first deliberate production choice; B3-64 is less stressful.

Scenario D — WebRTC-first, higher CPU pressure, future growth

Pick: B3-64 — only slightly more expensive than C3-32 while giving more RAM, more network, equivalent vCPU count, and better overall HoboStreamer balance.
